# Phase 15：三輪確認對話流程

## Context

原本 orchestrator 一次輸出所有內容（行程 + 住宿 + 交通），造成回覆過長、用戶難以逐段確認。Phase 15 將流程拆為三輪：先呈現行程 → 確認後提供住宿 → 確認後提供交通與預算，讓用戶有機會在每個關鍵決策點調整。

**實作類型：Prompt-only（不改程式碼）**

**先決條件**：Phase 10（結構化輸出）、Phase 11（預算計算）。

---

## 三輪對話設計

```
Turn 1 — 行程確認
  call_agent("attractions") → 呈現 Day-by-Day 表格 → ask_user("行程 OK？")

Turn 2 — 住宿確認（用戶確認後）
  call_agent("accommodation") → 呈現住宿比較表 → ask_user("住宿選好了？")

Turn 3 — 交通 + 預算（用戶確認後，最終輪）
  call_agent("transportation") → 呈現交通路線 + 預算 + 實用提示
```

---

## 階段偵測邏輯

LLM 透過讀取對話記錄判斷目前在哪個階段：

| 條件 | 行動 |
|------|------|
| attractions 資料尚未呈現 | Phase 1：呼叫 attractions agent |
| attractions 已呈現但用戶尚未確認 | 等待用戶回覆 |
| 用戶確認但 accommodation 尚未呈現 | Phase 2：呼叫 accommodation agent |
| accommodation 已呈現但用戶尚未確認 | 等待用戶回覆 |
| 用戶確認但 transportation 尚未呈現 | Phase 3：呼叫 transportation agent |

## 各輪輸出格式

### Turn 1 — 行程概覽

```markdown
## 📅 X 天 [目的地] 行程建議

| 天 | 區域 | 亮點 |
|----|------|------|
| Day 1 | 萬華 → 信義 | 龍山寺、台北 101 |
| Day 2 | 士林 | 故宮、士林夜市 |

[一句話行程主題說明。]
```

### Turn 2 — 住宿推薦

```markdown
## 🏨 住宿推薦

| 名稱 | 每晚價格 | 位置優勢 | 適合 |
|------|----------|----------|------|
| Hotel A | $60–80 | 台北車站旁 | 交通方便 |
| Hotel B | $120–150 | 大安區 | 安靜舒適 |

> 💡 [一句話選住宿建議]
```

### Turn 3 — 交通 + 預算（最終）

```markdown
## 🚌 交通安排

| 路線 | 方式 | 時間 | 費用 |
|------|------|------|------|
| 飯店 → 龍山寺 | 捷運板南線 | 15 分 | $1 |

## 💰 預估費用（X 人）

| 項目 | 費用 |
|------|------|
| **預估總計** | **$XX–$XX** |

## 💡 實用提示
- [2–3 條最重要的提示]
```

## 驗證方式

1. 輸入：「幫我規劃 5 天台北旅遊，2 人，預算 $1500，喜歡廟宇和夜市」
2. **Turn 1 確認**：只顯示行程表格 + 確認問句，無住宿/交通內容
3. 回覆「好的」→ **Turn 2 確認**：只顯示住宿比較表 + 確認問句
4. 回覆「確認」→ **Turn 3 確認**：顯示交通 + 預算 + 實用提示，無再追問
5. 確認每輪格式：有 emoji 標題、表格為主、無冗長段落
